2 · Profile data model

Every person has one profile. Forms read from and write to this record so data entered once is reused everywhere.

Account (auth)

  • Email · login identifier
  • Password · hashed, Firebase-managed
  • Role · Volunteer / Leader / Coordinator (default: Volunteer)
  • Status · In-process / Active / Paused / Inactive
  • Created at · auto-timestamp

Personal info (filled on first login)

  • Full legal name · First · Last
  • Preferred name (optional)
  • Date of birth
  • Phone (cell)
  • Street address · City · Province · Postal
  • Emergency contact · Name · Phone · Relationship

Ministry info (filled on first login or later)

  • Age groups willing to serve (multi-select: Nursery / Preschool / Elementary / Preteen / Youth)
  • Service times (multi-select: 9am / 11am / Wed PM / Special Events)
  • Spiritual journey / testimony (short paragraph)
  • How long attending Bethany Chapel

Compliance / audit trail (system-managed)

  • Covenant signed · date
  • Police check · date submitted, date cleared, expires on
  • References · two received
  • Training progress · per-module completion date + score
  • Renewal due date · auto-calculated (annual)

3 · Sign-up flows

Self sign-up (public)

1Land on siteNew visitor clicks "Sign up" on the sign-in page.
2Enter email + passwordFirebase sends verification email.
3Verify emailClick link in inbox. Account becomes active.
4Complete profileGuided multi-step form: personal, emergency contact, ministry prefs.
5Welcome dashboardStatus shows "In-process". Next steps: covenant, references, police check, training.

Invite flow (Coordinator-initiated)

1Coordinator sends inviteFrom Admin panel, enters volunteer's email + optional pre-assigned ministry.
2Volunteer receives email"You've been invited to join Bethany Chapel Safeguard Hub. Click to sign up."
3Invitee clicks linkOpens pre-filled sign-up (email locked to invited address).
4Sets password + completes profileSame flow as self-signup, starting at step 4.
5Coordinator notifiedDashboard shows new volunteer, nudges for covenant + police check.

5 · Technical stack

  • Firebase Authentication · email/password login, email verification, password reset, session management. Free.
  • Firestore · NoSQL database for user profiles, form submissions, training records. Free tier supports thousands of users.
  • Firebase Hosting (optional) · could replace GitHub Pages. Or keep GitHub Pages and just use Firebase for auth + DB.
  • Role-based access · enforced with Firestore Security Rules (server-side, tamper-proof).
  • Email invitations · Firebase sends transactional emails via built-in templates, no extra service needed.
Migration note — existing Google Sheets backend can be kept as a secondary export destination (for coordinator convenience) but Firestore becomes the primary source of truth. New forms write to both during transition.

6 · Claude Code handoff brief

Brief for Claude Code

Build a user authentication + role-based access system for the Safeguard Hub, a child-protection compliance site for Bethany Chapel. The current site is a static GitHub Pages deployment (github.com/Stophoto/safeguard-hub) with React-in-HTML forms submitting to Google Sheets via Apps Script.

Goals

  1. Add Firebase Auth (email/password) with email verification and password reset
  2. Add Firestore for user profiles + form submissions
  3. Implement three roles: Volunteer, Leader, Coordinator — enforced via Firestore Security Rules
  4. Build sign-in, sign-up (self), profile setup, volunteer dashboard, coordinator dashboard, invite flow
  5. Retrofit existing fillable forms (FRM-001, 006, 007, 012) to auto-fill from profile and attach user ID to submissions
  6. Keep Google Sheets backend as secondary export (don't break existing workflow)

Reference this document for: role permission matrix (§1), profile data model (§2), flow diagrams (§3), and wireframes (§4). The visual design should match the existing hub — navy (#1B3A4B), teal (#2E6B7F), gold (#B09055), cream (#FAF8F3), DM Serif Display + DM Sans.

Deliverables

  • Firebase project setup instructions for the owner
  • New auth pages (sign-in.html, sign-up.html, profile-setup.html, forgot-password.html)
  • Volunteer dashboard and coordinator dashboard pages
  • Updated index.html gated by auth state
  • Retrofitted fillable forms reading from user profile
  • Firestore Security Rules (repo'd and documented)
  • README update with auth setup and deployment steps
